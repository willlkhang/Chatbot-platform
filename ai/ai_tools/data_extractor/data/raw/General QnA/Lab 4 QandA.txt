Q:
I understand that Date class doesn't deal with string. However the Date is a data member of Result. So can Result handle the string for month. That is how I have got my lab 4 using a switch statement. Would this be ok or would I loose marks.

A:
No, it is not ok.

Both Date and and Result classes are data classes representing Model (in the Model-View-Controller design pattern) MVC. See https://developer.mozilla.org/en-US/docs/Glossary/MVC. As you can see this pattern is used not just in C++ development, as this link is for Mozilla's web development. Topic 2 readings also has MVC. Lab 2's exercise asks a specific question on MVC. 

By putting string into Result, you would be creating the same problem as putting string into Date. Both the Model and View are being combined. Doing that also violates the Single Responsibility Principle (the S in SOLID). Low cohesion.

Model and View are separate concerns, so need to be separate from each other.

A worked example using Java is found here: (has Student class) 
 https://www.geeksforgeeks.org/mvc-design-pattern/ 
 
 Can use the above example as a guide for later work, but for now follow the advice given where display string is not part of the model.
 
As the Date class serves as the Model, and how the data is viewed represents the View. The Model and View must not be coupled otherwise changes in View will require changes to the Date class. A number of students tend to use string to represent month in Date. That creates a representation problem that is coupled to human language (English, Thai, Korean, etc), and a cultural problem (US style date, vs other date styles). 

There is a processing problem. Date is an ordinal type. That means there is normal behaviour to compare date objects, Date objects can be equal, less than or greater than another date object. A month stored as in integer is an efficient compare on the CPU's ALU, whereas a string month requires comparison of a character array, one char at a time. Each char requires a fetch operation to get into the CPU.

The I/O from lab 4’s Registration exercise is the View component. You are removing the View component from the Model. It reduces the responsibility of Registration. It is an example of Separation of Concerns (SoC) software design principle. For details of SoC:
 https://www.oreilly.com/library/view/software-architects-handbook/9781788624060/8ff905c2-217a-47f0-85c2-789296d42e8d.xhtml#:~:text=Separation%20of%20Concerns%20(SoC)%20is,concerns%20as%20much%20as%20possible.
 https://www.geeksforgeeks.org/separation-of-concerns-soc/#:~:text=Separation%20of%20Concerns%20(SoC)%20is,and%20scalability%20of%20software%20systems.

 

The concepts don't just apply to lab 2, but everything in this unit, later units and the rest or your career after you graduate. So it is not just about loosing marks, but investment in your career. Q:
I am getting a lot of problems when using using const. When I remove all const the compiler lets me compile and I can run my program. Do i really need const.

A:
A method is declared const when you want to make sure that the method will not modify class data. It is design safety to ensure that data members don't get modified by mistake or ignorance. So const is used. It is a design feature where the designer is telling the compiler to enforce the rule that class data is not modifiable.

const is also used when you are passing a const reference to a routine or method. The const ensures that routine/method will not modify what is being passed in. The use of const is also part of the design and has to be used when the design calls for read only access. The compiler enforces the design rule.

When you declare something as const, the compiler holds you to your word. So if a method is declared const, it can only call other methods that are also const. Imagine a situation where the compiler permitted the calling of other methods that are not const - the other non-const method can change the data, thus violating the const declaration. 
This is explained in the textbook. You should also familarise yourself with the "Law of Demeter" https://en.wikipedia.org/wiki/Law_of_Demeter.

A const method should not be able to return a reference to class data. It should instead return a const reference. Get methods are usually const, so if the return of some internal data is using a function approach the return type must be a const reference. Doing it this way lets the compiler enforce the design rule to properly protect the internal data.

When a parameter is passed as a const, the requirement is that the parameter is read only. To enforce this, the compiler will ensure that only const methods can be called for this const parameter.

To avoid most problems, the rule to follow is getters are const, setters are not. The problems arise when you try to use a getter to get data inside an object so that you can call a setter on the internal data object to change the internal object. Does this approach make any sense at all? A getter should not be allowed to change the internal data and that is why getters are const. So the compiler is not going to allow a setter to be called on anything associated with a const getter. The compiler is only enforcing a safe design rule that is required by the class designer. For the work in this unit, you are the designer, so the compiler is only protecting you from your own mistakes when you start implementing your design.

So rethink the approach. Only setters change internal data - setters are non-const. Getters return the data - they are const. If you need to change some object that is buried deeply in some outer object, it is a chain of setter calls all the way down to the object to be changed. 

Objects returned by getters should not have setters applied to them. The word "const" on the right side of a getter method enforces this.

If there are 3 objects, a, b and c, if a contains b, b contains c, you know that for b to exist, c must exist, and for a to exist, b must exist. So construct c first using constructors and setters for c as needed, then construct b using constructors and setters and sending c in, or its data in to b, ... and so on.

Q:
I'm up to exercise 5 now for the assessed lab. I have converted the IO operators of registration from Topic 2 (still in C string) to not be friends as a test program before attempting it on my assessed-lab registration code classes. I have the operators as non-members to the class, and not as friends. 

I have it working, without throwing any errors and producing the output as desired. I would like some feedback regarding the "logic" of the code, as the interaction concept between the friends (or lack thereof) and the IO operators across the classes is very new to me. I'm not sure if I've designed the program properly and I'm only getting the correct output shown by an erroneous design pathway. 

A:
C++ allows operators to be treated as subroutines. That is the way it should be, as the execution of an operator results in execution of code behind the scene. C++ allows the operators to be defined for use on user defined types to keep the same semantics (meaning) as that when the operator is used for built in types. Details are in the Topic readings from the textbook "Chapter on Overloading and Templates". When an operator that operates on built in data types is used for user defined types, that operator is being "overloaded".

There are two usual ways that a subroutine (and that includes operators) can access a classes private data. There is a third approach, and this approach is bad - declare data members of a class as public. Unfortunately, you have taken the third approach. You would have learnt that this was bad in the prerequisite unit (ict167), but from the looks of it, this approach was taken to make the operators work, thus violating class encapsulation. It is a case of "out of the frying pan and into the fire". Remember a class provides strong encapsulation where by default everything is private.

The usual two approaches:
One way is the obvious way when the subroutine is a method of the class. Covered in the prerequisite unit (ict167) too.

The second approach is to make the subroutine (or even a whole class) a friend. In the second approach, the subroutine is just a normal function/procedure (like that done in ict159) with file scope. Normally such subroutines would need to use setters/getters to access private data of a class. But if a class makes the subroutine a friend, that subroutine does not need the public setters/getters of the class to access private data. Declaring friendship violates encapsulation of the class. There are situations (outside the scope of this unit) where we may declare friendship, but coding convenience is not one of them.

In the lab exercises, the operators are normal subroutines (non-methods) that have been given privileged access to class internals via friendship. For the lab, this privilege is to be removed by removing friendship. Of course, once friendship is removed, access to internals can only be made via the classes setters/getters. That is why exercise 1 asks for a full complement of setters/getters.

Q:
If I'm supposed to explicitly use the set and get methods instead of public variables in the IO operators, I have no idea how to call the set and get methods and use them in the context of a non-friend, non-member IO operator. 

A:
For input, you would need to read in the value from the file yourself (you write code to read this in). Values in files are basic data types - integer, string.. etc. Read into a temporary variable of one of the basic data types.

Then call the setter for the class with the temp variable as a parameter to the setter.

For output, use the relevant getter to get the data from the class. If the internal data of the class is a basic data type, you have the data you need to print out. If the internal data type is another class object, you call the relevant getter for that object. If that getter gives you another object, you call the relevant getter for that object. You do this until the final getter returns a value of a built in type that has the information that you after. This value can then be output.

In your code, you have data members declared public. This is wrong. Remove them and only keep the private ones. The approach is violating class encapsulation.

Your operators can still access data members of the class directly. Friendship was removed, and so direct access was denied by the compiler. The compiler did the right thing. You should not try to get around the compiler by then declaring the data members public.
For example, this is wrong:
istream & operator >>( istream & input, Course & C )
{
    input >> C.sectionpub >> C.creditspub;   <--this is direct access of private data for Course object
    
	return input;
}

By declaring the data members public to make the operators work, you have made the situation worse in terms of class design.

Follow the approach that I have indicated above. Declare two temp variables inside the operator body. Read into those variables. Then call each setter for Course object C with the appropriate temp variable as parameter.

Once you have removed friendship for all operators, move all the operators to the file main.cpp so that anyone looking at the code for Course or Registration will not know that these operators exist. 

See Rule 44 Prefer writing nonmember nonfriend functions in the unit's reference book: C++ Coding Standards: 101 Rules, Guidelines, and Best Practices by Alexandrescu et al. Ebook is in the library (My Unit Readings) if the following link does not work. http://lib.myilibrary.com.libproxy.murdoch.edu.au/Open.aspx?id=291948

A:
There are a few issues. The primary issue is that you have gone 
backwards. What is asked for is to make the operators non-friends and 
non-members. The code shows that the operator has become members of 
their respective classes.

A::b means b belongs to A. See the various usages of the scope resolution operator :: in the textbook.

For the other issues, I have labelled the lines as L1, L2 .. etc

L1 is ok.  float is a built in type, so the operator can do its job

L2 should not have a return type as void. Otherwise you can't chain 
read into different variables like you might use with cin

e.g. can't do the following as return type is void
cin >> a >> b >> c; // to read into 3 variables. cin is used in this 
example but cin can be substituted with another input stream object.

L3 wouldn't compile
For it to compile, an operator>> is needed where the left parameter is an 
istream reference and the right parameter is an object of class Result. 
There is no such operator.

The code that you have (supposedly to read into a Result object)
istream & Result::operator >>( istream & in)

is saying that the left side of the binary operator is a Result 
object because you have Result::operator >>(..). This says that the 
operator is a member of Result. The right side is an istream 
reference. How this is the case is something for an another day as it 
will detract from the current discussion. Binary operators belong to 
the left operand. So to be able to use his operator, L3 needs to be:

res >> in;

which is a very odd way of doing input into the res object, but given 
the way the operator is declared, it is the only way to call it. The 
compiler is not trying to be smart, It is just following simple rules.

The same problem is there in main where a registration object is being read in using the operator.

Q:
...have worked my way up to Question 5 of Lab 4 where the I/O operators are to no longer be friends or methods of classes and students are to use their set and get methods instead.

Does this mean that students are effectively removing the operator overloading methods from the classes entirely and using set/get methods in their place?

A:
The operator overloading is still there as the operator code body is provided as the operator is being used in the main program. The difference is that the operators do not have direct access to the class internals. 

A class is a mechanism for encapsulating. Any method is a hole in that encapsulation. But you can't do without methods as that would make the class un-usable. 

So the compromise is to have only those methods that are needed to make the class functional - in terms of both class management (constructors, destructors -where needed, setters and getters), and behaviour (what exactly is the class for). This keeps the class minimal and aids cohesion. See how this relates to S and O in SOLID.

Do not provide constructors, destructors, etc. without a valid reason. Rule of 3 applies if class has pointer data.
