template <class T>
int Vector<T>::GetLength(){

There is very strong advice that the operator semantics shouldn't change when overloaded for use with user defined classes. This means that you are doing the right thing.

The [] operator allows set/get on existing array data. What that means, the operator lets you access array memory that already exists, or update data at an **existing** memory location. It should not allow access to memory outside itself. An example that I used with the students who asked me about this:

Q:
I have a general question. In what circumstances would it be appropriate to include the implementation in the header, for a class? Such as Vector.h, that we have designed, what are the hallmarks which indicate that design approach should be taken?

A:
The purpose of separating the specification (aka headers, interface to the class in your question) from the implementation is so that users of the specs (the what) don't create dependencies (in their application) on the way the implementation (the how) is carried out. Clients just want to use the interface to the class to get their job done. If the implementation is changed without changing the interface, perhaps to increase efficiencies, the applications don't break as the application only depends on the public specs. This means that the client only depends on the specifications (the what) and not the implementation (the how). The implementation needs to be tucked out of sight.

In the Registration exercise of Lab 3 and 4, you were changing the implementation, but the client code in main() wasn't changed.

This separation in design must always be followed unless there are very good reasons for not following. See more in Topic 1 and later readings. Summary of readings is in the file "Essential Reading List by Topics File" in LMS.

There are situations where non separation can be defensible.

1. profiling data shows execution speed boost is needed - some code is inlined. There needs to be actual profiling evidence before this option is taken. Usual cases would be pass through setters and getters. You know about inline code as you have seen it in Registration exercises. For the Registration exercises there was no good reason to inline so you had to move the implementation to the .cpp files.

What does the coding standards book that we use as reference have to say about premature optimisation?

If you want to find out more about inline functions (not just inline methods), including pros/cons with examples, see 
https://www.geeksforgeeks.org/inline-functions-cpp/

The link above also gives advice about pitfalls of using macros, and if you are planning to use a macro, use an inline function instead as the compiler can do the necessary checks.

2. It is not possible to do separate compilation as the type is not known - template classes. In your example of Vector, Vector.cpp can't be compiled to Vector.o (or Vector.obj) as the type information is not available in the Vector class on account of it being a parametrised type.

In both the above cases, you still tuck the implementation (the how) out of sight below the the specification. There is the additional advantage that it makes the specs less cluttered and easier to read by the user.

To use an everyday example, to drive a car, you get in the car and use the interface provided by the car (steering wheel, accelerator, brakes, etc). The interfaces just convey what can be done with the car. You don't want to have to deal with how each of these things work - actually see them working while driving. If an engine is changed to a more efficient engine, the interface doesn't change so you still drive the car as before. Or, if you get into a different car (make/model), the steering wheel function has to remain the same.


Q:
I do have an important question to ask this time, and I hope to seek your guidance about it.
 
It's about accessors getters/ setters, and whether they break the rule of encapsulation resulting in poor OO design.

I have tried to read and search multiple resources about this, but I have yet to come with a clear answer, with different resources and articles giving different opinions on Accessors.

From my understanding, encapsulation hides the internal representation, state of the object from the outside. So the users of the class should not even know about things that are private. However, don't setters and getters contradict encapsulation and even ruin it, because it exposes the private variable by letting the user know about the private variable existing. Whether you use get/set to control what the user read and write does not matter here, because the user should not even know the private variables exist in the first place. 

It also contradicts how an object should actually behave. From my understanding, an object is supposed to represent things as a whole. Like a Dog object you have used as example. It has some intelligence as if its a real thing. with methods like bark etc to represent a behaviour. Whereas as compared to the time and date class used in lab 4/5. It uses classes like a bag to hold things and used to only retrieve part of the data such as an hour, min and seconds. With this case, you might as well use a struct to hold data. Because you only are retrieving part of the data and not as a whole, unless you are using the time class to return hour min and seconds as a whole. 

However, even so,  with the only set and get for everything in a class seems like a bad design to me and does not encapsulate the variables since everything is exposed to the user in my view. 

A:
The encapsulation controls access to the internals, and the only thing that a user needs to know is "what" the class does, not "how".

By declaring anything as private, direct access to that data/method is prevented. Access to the private data is now only possible via a public method. The public method can impose restrictions on what are valid values. If you use a struct to encapsulate (a struct is also an encapsulation), you as a designer are indicating to the user that there is no access control to what is valid or not valid.  A struct is like the bag you are referring to. So if you have time as a struct, the user can set seconds to be >=60 directly and it will remain that way. If you control the range of values by the class mechanism, then you can restrict how seconds can be set. For example, the user can, through a set method, set seconds to be 120 and the setter can allow this, but the setter would also convert that to 2 min and 0 sec internally.  That is internal behaviour that keeps the time object normalised. The user doesn't have to worry about the details, the object takes care of all that.

So with the class mechanism, it doesn't mean that you are trying to hide that a Time object has hours, minutes and seconds. The user needs to know about hours, minutes, and seconds, but doesn't have to worry about the details of how hours, minutes and seconds are handled. How these values are stored is also not relevant to the user. All the user is interested in is that the Time object is efficient and does what is expected of Time objects and has access to the components (hour, minutes, seconds) of Time.

Using a different example, lets say a car. The driver of the car knows that there is an engine. The driver doesn't keep opening the bonnet to see how the engine is working but views engine parameters (getters) and controls (setters) the engine via the interface available from the driver's seat. 

If the designer is really paranoid about information hiding, the designer can use the PIMPL idiom to totally hide the internal data structure but this approach can be extreme. There is advice about this in this unit. See the 101 Coding Standards reference book we use, Rule 43.

Encapsulation also includes separation of the class interface/declaration/specification (what) from its implementation (how) - file level encapsulation. Non-template classes have the implementation (how) in implementation files (.cpp for C++, .c for C). Class interface/declaration/specification (what) is found in specification files (.h for C++ and C).  This separation leads to abstraction - know what the class does, but not how (details).

So if you used Date and Time classes as a "bag", you know that your OO design needs to be improved. You are not hiding the fact that Date has day, month, year as the user of the class needs to know about day, month, year. Certain day, month and year combinations are not valid, and a bag like structure would allow invalid dates. 

See See the 101 Coding Standards reference book rules 11, 41.

Rule 41 has further advice on setters and getters.


Q:
Based on what we've learned I'd argue that putting file and standard I/O together in a single "General I/O" class would contradict the S from SOLID, and that separate classes is preferable for different types of I/O. But I just want check with you on this: Are there any reasons to have standard I/O and File I/O in a single class?

A:
I am assuming you mean keyboard/screen by the word "standard".

The answer to this was decided many decades ago, before there was a language called 
C++, or Java, or C#...

At a high level (abstract level), I/O is flow of data to/from the processing area (your program) and outside. "Outside" means from external environment, keyboard, files, screen console., etc. The specifics vary, in that the source (input) could be a keyboard or it could be a file and the destination (output) can be screen or file.

So the modelling of I/O is done using streams. As far as the processing is concerned, it is looking at input streams or output streams. What the stream is connected to in the outside environment is not really relevant as the algorithm implementations only process the data.  Using this approach, the processing environment is not made dependent on the actual source/destination. So the processing environment doesn't have to change when new forms of source or destinations are being used as it only sees streams. All that is needed is for the programmer to connect the streams to the physical source/destination. 

The idea is that the processing environment (algorithm implementations) sees only the abstraction and this prevents the coupling to specific external hardware. This is an important idea. Java really pushes this, as Java code runs on a virtual machine and so Java programmers are insulated from the hardware. Java programs are written to abstractions of the various hardware, keyboards, screens, graphics, networking,, .etc. The hardware specifics may change but that is meant only to be seen by the virtual machine that runs the Java programmers code.

Classes model behaviour and transferring data is a behaviour. The model used has to be at the highest level of abstraction, so that user programs written to the highest level of abstraction don't have to be modified when lower level implementation changes. So you will see that the I/O stream library is built using an inheritance hierarchy that models data streams. http://www.cplusplus.com/reference/iolibrary/

So if you move away from this abstraction, you are not having more cohesion, you are just proliferating your code with special cases and creating couplings where these couplings are not needed. You are also reducing cohesion as the Input or Output behaviour is now fragmented in multiple classes that each deal only with specific external environments. **The cohesion applies to the behaviour and classes are known by their public behaviour.**

The may be good reasons to model a keyboard, (or some other device) if the application is to build a simulation of a keyboard. You can have a keyboard class then. In most cases, the processing environment (algorithm implementations) is dealing with data.

Q:
I have my overloaded extraction and insertion operators for each of the class in main.cpp. Do I need to doxygen comment them?

A:
Everything inside the class declaration has doxygen comments. Typically in .h files. See example below.

Everything outside, normal comments.

So main is outside a class declaration, use normal comments.

**  File A.h  **
// relevant #include only

/// START OF DOXYGEN COMMENTS, even for template classes
class A   /// start of class A declaration 
{
	// Class methods are found here.
	// Only the first line (prototypes of the methods are here)
	// Method bodies (implementation) are not found here
	
	// Unlike Java, C++ classes will not have method bodies here
	// unless you have very strong justification for having it here.
	// There can be heavy penalties for doing it the "Java way".
	// Java has only one view of the world - all objects
	// C++ allows choice and therefore choice needs to exercised carefully and deliberately

	/// Inside the class declaration - doxygen comments
	/// doxygen comments indicate what the method does, **NOT how**.
	/// It is a description of the behaviour, *not how* this behaviour is achieved.
	/// Parameter descriptions and return data description just says what the
	/// data is for, or what the data represents.
	/// You should use the file modelfile.h in Practice/Doxyexample to guide
	/// you with doxygen comments.

	
}; /// end of class A declaration. END OF DOXYGEN COMMENTS, even for template classes

// outside the class, but still in A.h - normal comments. NO doxygen comments here
// the situation with template class files where you will have implementation in the .h file

**  File A.cpp  **
	// Implementation - (body of the methods)
	// this is outside of class declaration - NO doxygen comments,
	// just normal comments
	
	// Template classes would not have a .cpp file.
	// The method bodies (implementation) would be in the .h file after "end of class 
	// declaration". Normal comments used in the implementation.

** File main.cpp  **
// No doxygen comments here
// This is not a class - it is where the main() routine lives
// This is the application program that makes use of the classes.
// So normal comments

The doxgen comments are used to produce external documentation.
The doxygen program uses the comments to produce external documentation.
For our case, this documentation is in html.

If you have used Java, which all of you would have from prior studies, javadocs is used to produce external documentation from javadoc comments inside your .java files. Doxygen does the same thing (as javadocs) for C++ files.

I also use Doxygen to analyse structure of large codebases.

Q:
I've been talking to some of the people in my class and when they run

A:
It doesn't have to be double. Other strategies can be used. The main thing is to strike a balance between allocating memory when needed (processing needed) and the amount of memory used (and left empty, thus wasted). One approach is in the lecture notes. There are others. The idea is that over the lifetime of the Vector, the average  (a more accurate term is "amortised") time for inserting at the end of the Vector is a constant value. The actual multiplier by which to increase is implementation dependant. See complexity of push_back http://www.cplusplus.com/reference/vector/vector/push_back/. The discussion at this link is for the STL vector. You can't use the STL vector in your assignment as you need to write your own from scratch. The discussion at the link is still relevant.

For the STL vector, I have come across multipliers of 1.5 and 2. VS2010 uses 1.5. The STL vector can start with no memory. Do not use multiplies larger than 2 as it will over the lifetime of the vector usage waste memory. There is a need to balance memory space and time to allocate and then copy elements. Some discussion about this was made some time ago by Andrew Koenig and Barbara Moo at http://www.drdobbs.com/c-made-easier-how-vectors-grow/184401375. Koenig was at the same lab as Stroustrup, the creator of C++. Both Koenig and Barbara are acknowledged experts in the field. In a discussion forum some years ago, Koenig worked out the value of 1.5.

q:
This has made me wonder:

Is there anything special about an array, or is it just a 
contiguous series of data of the same type stored in memory? 

a:
Nothing special, just contiguous memory. At the level of contiguous 
bytes, the type isn't even relevant.

In the example code (b) that you have for creating array:

Once you have created the array, of let say, T binds to integer (i.e. int), making array an integer array of arraysize,  and assuming 32 bit ints, I can do this,

char *dodgyC = (char *) array;

dodgyC[0] is the first byte of the first 4 byte int value of the array you created.

The pointer type is the "lens" or "window" through which the contiguous series of bytes are viewed when the pointer is dereferenced. [] does pointer arithmetic and then a deference. So array[2] is actually *(array+2). array+2 is the pointer arithmetic (addition) and it is adds 2*sizeof(pointer type) to array.

Assuming a double is 8 bytes (64 bits) on your particular implementation,

double *dodgyD = (double *) array;

dodgyD[0] consists of 2 int values (2 x 32 bit ints) from the array but accessed as one element via dodgyD and represented as a 64 bit float - yes dodgy indeed. The two int values are accessed as one element via dodgyD[0] because dodgyD is a double pointer.

When accessing memory via a pointer, there is no way to see memory as anything else than the type of the pointer. If you are using an int pointer, you can only see int. If you use a char pointer, you will see char, ... and so on.

See description of the various builtin types in the textbook or 
https://en.cppreference.com/w/cpp/language/types

Pointers are powerful, useful but *very dangerous* mechanism. Memory access can happen in unintended ways. You will be experiencing it first-hand, instead of just reading about it, as we are using a powerful programming language. 

You may have heard of what happened in what is called the "largest IT outage in history" when a security company called CrowStrike pushed an improperly checked update.

A memory access violation at the privileged level can have very bad consequences.
 
CrowdStrike own analysis indicates that the Sensor is written in C++. There was a pointer to an array of 20 elements. It was such a basic error that people learning to program make, trying to access the 21st element - an out of bounds access.  https://www.crowdstrike.com/wp-content/uploads/2024/08/Channel-File-291-Incident-Root-Cause-Analysis-08.06.2024.pdf 

You will be making similar errors in this and later labs and learning from them. For you it is just loss of marks as your code would not be running at a privileged level and the OS will just kill your program. But the consequences when this error happens in production code is staggering. The amount of money that will be finally be involved would be eye-watering.

https://ia.acs.org.au/article/2024/crowdstrike-sued-by-shareholders-over-mass-outage.html

The OS allowed the memory access in privileged mode and damaged itself. There is another lesson to be learnt. Zero trust principle is used in security, but it should also be used in software architecture. 

The other concern that there is with pointers is that when a data structure like the Vector you are building contains pointer to data. Then the rule of 3 applies. See Topic 5, Lec-13.ppt. The required methods must be tested to make sure that the data structure is useable. 

q:
Is it ok, that I've got pushBack() and popBack() methods in my Vector template?

a:
What would happen if you needed to put a value in a specified location - like the way an array is supposed to behave? i.e random access as opposed to sequential access.

You would need to provide a method or operator for this access. You would already have a method to return size. Once you provide this random access method, you have duplication of functionality and this is not good from a design point of view. It is not minimal any more. 

There is a (very good) case for pushback as access using operator[] doesn't increase size and the [] access enforces bounds strictly. Pushback is needed to keep putting new data into your Vector. In this situation operator [] is used to set/get existing array elements and so there is no duplication by pushback. See Rule 80 in coding standards.
