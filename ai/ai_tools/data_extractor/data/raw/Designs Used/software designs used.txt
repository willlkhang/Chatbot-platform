Q:
For my date and time classes I have kept the input and output operators in date.h and time.h. You had told me to remove them as putting the operators in the .h files violated MVC. I think I understood what you told me, but now I think I am not so clear as I didn't write it down.

A:
I/O is tied to formatting. That is what the input data or output data looks like. It is part of the "View". The model is the actual data class, Date and Time for this work. You need to uncouple the View from the Model. Remember what you would have done in first year about "Coupling" (and also Cohesion). 

If the Model is coupled to the View, if the View needs to be changed, the Model has to be changed resulting in large code changes throughout the code base. The assignment in this unit is tiny compared to code bases out there, but the principles still apply and you must always follow the principle where you minimise coupling. All this was discussed in the workshops. 

If you keep the View with the model, you will get no marks. Not getting marks is really a small price to pay compared to what would happen with industry code where a small change in the View in a poorly designed code base results in changes proliferating throughout the entire code base (Jelly effect referred to in the PowerPoint slide and workshops), leading to bugs, cost overruns, delay in delivery.

Lab 4 was about removing the View, including the formatting concerns from the Model, and the lessons must be utilised and built upon moving forwards.

Minimising coupling and increasing cohesion rules must never be violated. You would know this from ict159 last year where a mark of 0 would be given for the entire assignment if modularity rules were not followed. This is coming from industry as we become more industry focussed. 

So do not violate SOLID. The cost is heavy and it does not matter if the code works now. The problem will occur later.
