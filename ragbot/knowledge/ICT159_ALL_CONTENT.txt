• Inheritance
- One of the most useful aspects of object-oriented programming is code reusability.
- Inheritance is the process of forming a new class (called derived class) from an existing
class (called as base or parent class)
• The derived class inherits all the properties (data, methods) of the base/parent class
• It adds additional data and methods/functionalities that are specific to the new inherited class
- This is a very important concept of object-oriented programming since this feature
helps to reduce the code size.

Basics of O-O Design 
Writing a program in a structured, procedural language like C largely involves working out what 
steps you need to perform. Applying a process like top-down design, simply breaks those steps 
up into a hierarchy. Therefore, structured design focuses on the code and what functions need to 
exist to solve a problem. 
 
Designing a program in an O-O language is quite different. The design process involves looking 
at the data involved in the problem, and then working out what needs to be done in processing 
that data. Therefore, O-O design is very data-centric. 
 
Writing a program in an O-O language therefore generally involves: 
• Defining all the classes that are needed to solve the problem. 
• Creating instances of those classes (objects) that are needed. 
• Writing code to execute the code (called methods) that are part of these objects.  
 
Think about a simple program like the triangle program: 
• What data would this programme need to store? 
• What processing needs to be done on that data? 
• What other methods might be needed? 
 
Summary 
This has just been a very basic introduction to the concepts involved in object -oriented 
programming. 
 
If you go on to do more computer science, you will study object-oriented languages and O-O 
design. 
 
You would not have to apply O-O concepts in any of the programming for this unit. However, 
remember that the theory of these concepts might be examinable!

Building Classes 
Methods and Constructors 
To do OOP we need to first define what classes are needed. Classes are made up of code and 
data. 
 
Code modules in object-oriented languages are generally called methods (similar to functions in 
C). So, the code that makes up a class are all part of methods. 
 
The methods found in classes perform various operations that relate to what the class is for. 
Specifically, they perform operations on the data that is part of the class. 
 
However, there is also a general category of methods that are involved in setting up each 
instance of the class (object) for use. Whenever a new instance of a class is created, one of these 
methods is called to make it ready for use. This special type of methods is called a  constructor. 
 
Constructors look just like ordinary methods except they have the same name as the class 
which they are constructing. Note that since there can be a number of different ways to initialise 
new objects, there is often more than one constructor for each class. 
 
Creating a new object involves calling the constructor, which sets up all of the data inside of the 
new object and reduces the work that the programmer needs to do.  
 
Access Methods 
As indicated previously, the data inside of an object is generally not accessible outside of that 
object. To access these private data members, the programmer using the class must call a 
purpose made (publicly available) method. These methods are known as access methods. 
 
There are two basic categories of access method. 
 
Set methods allow the programmer to change the value within the object.  
 
Get methods allow the programmer to retrieve a value from within the object.  
• This could be a value that has been calculated by the method 
• Or it could simply be the value of the private data member, returned by the method. 
• The programmer won’t necessarily know which applies, which enforces encapsulation 
and abstraction. 
 
Using access methods allows the designer of the class to control how the programmer interacts 
with that class. For example, the programmer cannot change the values of private data 
members, except through an access method. 
 
The access method can then impose restrictions on what can be done.  For example, a Get 
method might refuse to give the programmer a value if the variables concerned don’t yet have 
values in them. Similarly, a Set method might prevent the setting of invalid values.

Modular Programming Principles and OOP 
We have previously looked at a number of principles relating to modular programming, namely 
abstraction, code re-use, high cohesion and low coupling. In general, these principles apply very 
strongly to OOP as well. 
 
Most O-O languages provide a number of features to enhance and enforce abstraction. These 
mean that people who are using classes do not need to know the details of how those classes 
work, just what they do. 
 
In OOP this is commonly called encapsulation. In fact, OOP enforces abstraction even more 
rigorously than non-OO programming. 
 
Access to either the code or data inside of an object is often restricted.  In particular, data inside 
of an object is rarely accessible outside of that object. 
 
These are referred to as private data members. However, code modules (functions) usually are 
accessible. Therefore, access to the private data inside of an object must instead occur through 
public functions. 
 
The designer of the class specifies which pieces of data/modules are accessible, and which are  
not. This allows control over how the program using the class can interact with it, and thereby 
help ensure it is used correctly. 
 
Code-reuse is a critical part of OOP in that the classes created must be general “blueprints”.  
• Objects can then be created as instances of these classes facilitating code re-use. 
• In more advanced OOP you can actually take an existing class and extend it by adding 
new features and adapting it to your specific requirements without needing to start from 
scratch. 
• This is called inheritance and allows even more code re-use. 
• Some O-O languages can also support the ability to apply a module to different 
combinations of types of data given as parameters. 
• This is called polymorphism and can also assist in reuse. 
 
High cohesion applies to both the design of the classes and their code modules/methods. 
• Classes only store data that is relevant to their task. 
• Methods are often particularly narrowly focused in terms of the tasks they perform.  
 
Low coupling also applies to OOP. 
• In some ways it is slightly less significant since less data is passed around between 
modules. 
• However, in others it is more important as data is generally completely inaccessible 
outside of the object.

Object = Code + Data? 
In OOP the objects are made up of both code (things that the object can do) and data 
(information about that particular object). In this way OOP bundles together both the operations 
that relate to the object and the information that has to be stored in order to perform those 
operations. 
 
This doesn't happen in procedural structured programming because the actual data doesn't 
really belong anywhere. 
 
So, our Person object might store data like the person's name, their phone number and their 
email address. This makes it quite a lot like a struct in C. 
 
However, unlike a struct the code to process that data is bundled together with it. 
 
So, there might be methods (functions in C) in an object to set the person's name to a given 
value and/or to send an email to them. 
• An object is a little like a struct that also has functions (methods) as part of it to perform 
operations on the data it stores. 
• These operations depend on what is appropriate to the object and what the program 
needs to do. 
 
Classes vs Objects 
So, an object is made up of code and data and, in OOP, programs are made up of objects.  
However, object-oriented code is often said to be made up of classes. 
 
The difference between classes and objects is actually a little subtle but is crucial to understand.  
 
If we have a Person object, then that stores data about a specific person and can perform various 
operations on the data relating to that person. However, there may be lots of different people 
for which data has to be stored so separate objects will be needed for each.  
 
A class therefore is a general specification of an object (i.e. a description of what an object of the 
class stores and what operations can be performed); you could think of a class as a blueprint or 
a design. We can write a Person class, and this won’t actually store data about any particular 
person, but rather defines what data can be stored and also what operations can be performed 
on an object of the class. 
 
We can then create any number of Person objects based on this Person class. This is a little bit 
like the data type int and declaring a specific variable of type int. We say that an object is an 
instance of a particular class. 
 
Classes are therefore effectively complex data types and objects are like variables declared from 
that type. The difference between variables and objects though is that objects contain code as 
well as data and can perform operations on the data they hold.

The Object-Oriented Paradigm 
Object-oriented programming can be done in the C language. However, the language was not 
designed for this paradigm and thus involves use of some fairly advanced features. Therefore, in 
this unit, we will only be looking at the theory of object-oriented programming. 
 
The Structured Model 
In a procedural program we have separate modules of code (functions in C) which do various 
things. 
 
We also have data which is declared inside modules and can then be passed around between 
modules. However, the data doesn't actually belong to any of these functions, although the 
variables that store it do. Instead the data is “just passing through”. 
 
As we end up with more and more complex data structures to model the problems we are trying 
to solve, there is more and more work maintaining and organising these structures. Therefore, 
our program will include a large number of functions that need to be called in order to ensure 
these data structures work correctly with the program. 
 
Programs are Made of Objects 
Procedural structured programming that is used in languages like C involve taking the high -level 
algorithm which solves a problem and dividing it up into the individual steps.  Each of these steps 
becomes a separate module in our program. Often these steps are broken up further into sub-
steps which means that a function will call other functions in order to complete its task.  
 
With the O-O paradigm, the program is divided up into objects. 
• These sometimes relate to physical objects. Eg: a program managing data about people 
might have a Person object. 
• Of course, they don't have to be physical things but by making an object for them they 
can be treated a little like they are. 
 
One major difference here is that O-O is much more data-centric in terms of developing a 
solution whereas the traditional approach is much more concerned with the steps involved in 
solving the problem (code-centric). O-O programs also contain code to do things with the objects 
otherwise they couldn't do anything.

Structures slides.pdf : CONTENT: ICT159 – Foundations of Programming
• Data Structure refers to data organization, management, and storage format 
that enables efficient access and modification
• We have seen so far two types of data structures
- Simple variables, which can be of various types (int, float, char, short, byte, etc.)

Things to Note: 
• The name variable in struct item is declared as a pointer, without any size. Allocation 
of memory for the string is done in FillBin(). 
• The allocation of memory uses the calloc function belonging to the stdlib.h library. 
This function returns a char * as name is declared as a char *. You should look up 
the calloc function online to understand its signature and usage. 
• As itemType is a pass by pointer parameter, the struct must be de-referenced inside 
both functions in order to access each member. This is achieved by de-referencing the 
struct first; we use parenthesis to force this operation first; That is, (*bin). We can then 
use the field selector/dot notation to access the required field. Eg: (*bin).stock. 
• Now you understand what is happening with that syntax, you will be happy to know that 
there is a handy abbreviation; the arrow operator. So, bin->stock is the same as 
(*bin).stock. The arrow operator is used in the above example. 
• In the function FillBin(), scanf() must be given the address of each item in the 
struct. Thus, the weird syntax &(bin->stock). Here bin->stock is enclosed in parenthesis 
and the address-of operator immediately precedes this. However, this is not required for 
the name variable, because name is a string. 
 
Combining Arrays and Structs 
The fields in a struct can be made up of any data type that exists at that point. As with the last 
example, this can include other complex data types such as arrays/strings, and even other 
structs. The important point to remember is that when accessed using the field selector operator 
(.), a field in a struct behaves exactly the same as an ordinary variable of that data type. 
 
However, you can also combine arrays with structs another way by having an array of structs. In 
the example before, we declared only a single variable of type itemType, so we can only store 
information about one stock item. More realistically there would undoubtedly be many such 
items stored in the warehouse. Instead of declaring individual struct variables we can create an 
array of such variables. 
 
Creating an array of structs is not very difficult, once the struct type itself has been declared. For 
example, the following code creates an array of 20 items stored in the warehouse: 
 
const int MAX_ITEMS = 20; 
... 
itemType bins[MAX_ITEMS]; 
 
When working with the array of structs, the same principle as always applies: 
 
Reduce whatever structure you're working with down to a simple type and then process this as normal. 
 
For example, the field stock within each element of the bins array created above is just an int. 
 
To access this integer, just use the square brackets ([]) to access the array element you want and 
then the field selector operator (.) to access the individual field. 
 
Therefore, to access the stock field in the first element of the bins array, you could write 
something like: 
 
bins[0].stock = 100; // accessing the first element in bins array

scanf("%d%*c", &(bin->lwm_rop));   // note address-of operator

Usage 
Now we can create our own data structures, how do we go about using them? 
 
Essentially, while the struct can be dealt with as a whole entity, many operations can only be 
performed on the fields that make up the struct. This is because these fields are primitive data 
types, which can be processed by the language. 
 
To access a field in a struct, we use the following ‘field selector’ or ‘dot notation’ syntax: 
 
variable.field 
 
So, if we have a variable called firstbin of our new type itemType, we can use the following 
assignment statements to put values into the struct fields: 
 
firstbin.stock = 10; 
firstbin.lwm_rop = 2; 
firstbin.value = 29.95; 
 
As these fields are primitive data types, the operations are simple. However, as the name field is 
a string, we would need to do something different. Eg: 
 
if(NAME_SIZE >= (strlen(“iPad Air”) + 1))// if enough memory 
strcpy(firstbin.name, “iPad Air”); // use library to copy string 
else 
printf("Not enough room to copy!\n"); 
 
Note that when using the field selector or dot notation, the expression has the type of the field 
being referenced immediately preceding the dot and the field immediately following the dot. For 
example, firstbin.stock. As the struct member stock is of type int, firstbin.stock 
behaves exactly like an ordinary integer. 
 
Parameter Passing and Complex Data Types 
In general, the same rules apply when passing complex data types into functions as those for 
passing primitive data types. One important principle though is that all complex types (including 
arrays and structs) should always be passed by pointer. This applies even when they are not 
to be modified. 
 
The reason for this is that complex data types take up a lot more memory space than individual 
primitive types (for obvious reasons). Although we have used a simple example where the 
contents of the struct are not huge (in memory usage), some problems (thus structs) can 
become much bigger. 
 
Making a copy of the entire complex data type (as would be done if passed by value) is therefore 
very often inefficient, both in terms of the memory space and the computational work required. 
So as a general rule, complex data types should always be passed by pointer.

Data Structures 
Structuring Data 
Arrays are extremely powerful constructs that allow you to solve problems that otherwise would 
not be feasible. However, they are not appropriate in every circumstance. 
 
To explain, if you store many instances of a single value, arrays work well. For example, if you 
want to store 100 exam marks in order to process them. However, if you want to store many 
instances of multiple values (i.e. of different data types), then arrays are not well suited. For 
example, if you want to store 100 exam marks along with the names of the students that scored 
each result. 
 
To solve these problems involving many different possible arrangements of data, you need to 
design your own data type. When we bundle together different data types to solve such a 
problem, we use a construct called a data structure. This essentially involves designing your own 
data type. 
 
User-Defined Types 
A first step towards being able to create your own data types is to give user-defined names to 
any defined data type. C provides the typedef statement that allows for this operation. 
 
For example, say we want to define a counter variable type, we might say: 
 
typedef int ctr; 
 
Now we can define variables of type ctr as if it were a built-in data type: 
 
ctr i; 
 
... 
 
for(i = 0;i < MAX;i++) 
{ 
    ... 
} 
 
The advantage of this is that the abstraction hides details like the exact type of the data while 
simultaneously telling the reader what the variable’s purpose is. However, simply re-naming data 
types is a trivial usage of the typedef statement and there are much more important uses. 
 
Creating and Using Data Structures 
Creating 
One common way that new data types are created in C is through something called a struct. 
This is essentially an abbreviation of the term data structure and is created by combining simple 
data types. 
 
While an array allows you to bundle together multiple pieces of data (of the same data type) into 
one variable, a data structure represents a new data type that you can define. From the new data 
type, you can create your own variables of this type.

Summary 
Files are useful as there are many situations where a program needs to deal with a lot of data ,

Files allow a program to store large amounts of data which remains on the disk after the

Files can be opened for reading, writing and appending only. This means that apart from

Algorithmically there are two basic ways of dealing with file data and these are:  
• To read the entire contents into memory all at once for processing. 
• To process only a single record at a time.

Algorithmic Approaches 
All in all, dealing with the large amounts of data that can be held in a file can be quite tricky.  
There are two fundamental approaches possible: 
• Multiple Loaded Records 
• Single Record Batch Processing 
 
Multiple Loaded Records 
This involves loading multiple records from the file into program memory. Most of the time the 
entire file is loaded. 
1. This will involve allocating sufficient memory to hold all of these records. 
2. This will usually be done with an array (or some other data structure). 
 
Because of the potential problems with allocating so much memory, this is clearly not ideal. 
However, for many problems it is the only option. 
 
For example, sorting the contents of a file generally requires that the entire file be held in 
memory. Manipulating compressed data such as that contained in image files also requires that 
data be completely loaded into memory. 
 
Another common example of a program that does this is a text editor, like the one you have 
been using to write and edit your programs. 
• The text editor works by opening a file for reading and loads the entire contents into 
memory. 
• The user then makes changes to the contents of this version of the file in memory  – note 
that the original file on the disk is not touched at all during this process.  
• Once the changes have been made, the user then “saves” the file. 
• This means the editor now re-opens the file on the disk, this time for “writing”. 
• The file is now truncated; its contents are deleted. 
• The new version of the file (which contains the changes the user made) is now written 
back to the disk. 
 
Since the edits made by the user to the file can be to any part of the file and in any order , and 
because the only ways of accessing the file are read, write and append, the only way of working 
with the file is to hold its entire contents in memory while editing it. It simply isn't possible to 
make edits directly to individual parts of the file so there is no alternative but  to hold the entire 
file's data in memory all at once. 
 
Of course, some files are simply too big to be loaded into memory all at once.  For example, 
digital video is often many gigabytes in size. In this situation a specialised approach is required 
whereby the file is dealt with in separate portions or “chunks” at a time. 
 
Single Record Batch Processing 
Also known as "File Chunking". 
 
The alternative to loading multiple records is to only load as little of the file into memory at one 
time as possible. This typically involves reading only a single record in, processing it in memory 
and then reading in the next record. 
 
Because the records in the file are processed one after the other in order and typically no 
interaction with the user is required, this is often described as batch processing.

Handling File Processing Issues 
One of the aspects of dealing with files is that it is quite possible for the amount of  data in the 
file to be very large. This is particularly true when compared with the amount of data in terms of 
variables etc. that a program may declare and store in memory.  
 
So, there are a number of issues here with regards to storing data in memory tha t comes from a 
file. Obviously, it is not possible to declare a separate variable to hold each piece of data from a 
file so often an array (or some other data structure or combination of data structures) is used. 
 
However, some memory will already be in use by the program. For data from a file to be stored 
in memory by the program, that block of memory allocated to store file data must be contiguous 
(all in a single block) and the amount of contiguous memory available may be relatively small.  
 
Operating systems may impose limitations on how much memory can be utilised by a single 
process (running program) and this is hard to know in advance. 
 
There are other issues too. For example, the amount of memory required to hold the file data 
may be larger than the actual size of the file if the data is compressed (eg: image files).

The data which makes up each record can be broken down into fields. For example, there might 
be a field for the name of the stock item, another for the number of that item currently in stock, 
etc. 
 
Although files can be a collection of records, the structure of those records can differ 
enormously from one file to another (depending on the nature of the information being stored). 
 
File Formats and Text Files 
The format of data within the file itself is completely the responsibility of the programs that will 
read from, and write to, the file. Most programs have their own format for encoding and storing 
data which often only they understand. 
 
However, there is a generic standard format whereby the data within the file is essentially not 
encoded at all. Instead the data is simply stored as lines of text characters, usually just in plain 
ASCII (with space separated fields). These files are commonly referred to as text files. Another 
commonly used format is known as Comma Separated Values (CVS), which are similar to text files 
except each field is separated by a comma. These two formats are the ones we will be working 
with in this unit. 
 
Working with Text Files 
The records in text files are made up of lines which represent a string of characters followed by a 
newline character. In C this character is represented by '\n'. Note, the length of the records 
(the length of the line) is not fixed for text files – it just keeps going until a newline is found. 
 
To gain access to a file (including non-text files) the program must make a request via the 
operating system to open the file. Most programming languages provide built-in routines for 
doing this in a simple and convenient way. 
 
The attempt at opening a file will either succeed or fail. Failure may occur if, for example, the file 
does not exist or the OS denies access to it for security reasons, etc. 
 
When opening the file, the program must also specify the operations it wants to perform on the 
file once it is open. Files may be opened for reading, writing or appending. 
 
Reading simply means that data can be sequentially read out of the file, but no changes can be 
made to the file. Files can also be opened for writing so that data can be sequentially added to 
the file. 
 
Attempting to open a non-existent file for writing causes an empty file to be created into which 
data can be written. If a file that already exists is opened for writing, then the current contents of 
that file will be erased, and new data can then be written into this now empty file. 
 
Opening a file for appending allows the program to write data to the file but this time the existing 
data in the file is not erased and the new data is added to the end of the existing data 
(appended). 
 
As indicated, when dealing with text files data can only be read from or written to that file in  
sequential order. This means making minor edits and changes to a file can sometimes be 
awkward.

File I/O 
Consider a program that manages the ordering of stock for a large department store.  If the 
program has no record of what happened the last time it was run, then it will be virtually useless. 
For example, every time the program is run someone will need to input all the details concerning 
the stock in the store. 
 
This might include information about each item, stock availability, price information and where 
the item can be re-ordered from. If the store has 1,000 stock items and information about each 
item takes 10 seconds to be re-entered then, every time the program stops running it will take 
nearly three hours of constant typing before it can be re-started. 
 
This by itself is clearly unworkable. However, the problem becomes much worse when you 
consider that if the purpose of the program is to track stock then the program itself may be the 
only ‘reliable’ and up-to-date source of information about the store's stock in the first place!  
 
If the program crashes or there is a power failure, then the amount of data lost will be 
enormous. Clearly there is a need for programs to be able to maintain data, even when they are 
not executing. 
 
Files and File Systems 
This is done by writing the data to some form of ‘secondary storage’ such as a disk. 
 
The exact way that data is stored on the disk is determined by the operating system and is 
referred to as the ‘file system’. However, virtually all file systems work in a similar way whereby 
data relating to a particular thing is stored together in something called a file. 
 
Files themselves are stored within logical locations on the disk called  directories. The analogy is 
often drawn between directories and physical folders which store papers ’, so the term folder is 
often used. 
 
Directories themselves are usually arranged in a hierarchical tree structure, i.e. one within 
another. 
 
Files and directories are both given names, however, to uniquely refer to a specific file or 
directory, its location relative to the directories it is in must be given. This is known as the path to 
that file or directory. 
 
Records within Files 
Although the format of the data within files does differ, the contents of a file can still be viewed 
in the same general way. 
 
Files can be thought of as being made up of a number of records. Each record is of the same 
general structure and records are usually of the same size (although not always). For example, a 
file might contain data about the stock stored in a warehouse, with each record in the file 
containing information about one stock item.

Summary of Passing/Returning Values Into/Out of Functions 
There are ways to pass/return values into/out of a function: 
1. Out of only, using the function return value. This mechanism is convenient, and ideal for 
situations where the function needs to return one value only. 
2. Into only, using pass by value parameter variables. This mechanism allows values to be 
passed into a function via parameter variables, for use within the function. Modification 
of any parameter variable is allowed, but any changes to a parameter variable is not 
reflected in the original variable (outside the function). 
3. Into and Out of, using pass by pointer parameter variables. This mechanism also allows 
values to be passed into a function via parameter variables, for use within the function. 
Modification of any parameter variable is allowed, and any changes to a parameter 
variable is reflected in the original variable (outside the function). This then provides the 
means of returning multiple pieces of data from a function.

Pass by Pointer vs Return Values 
Pass by pointer and return values do similar things: 
• Both allow you to get data out of a function, which would otherwise be impossible due to 
the limitations on variable scope. 
• However, these are two completely different techniques. 
• Even though the result is the same, the way they work is quite different. 
• Also remember that a function can only return a single value (by its return statement) 
whereas as many parameters as you like can be passed by pointer. 
 
There are also significant differences in how you use each of these features. 
 
Rules for Pass by Pointer 
With pass by pointer it is very easy. You simply put an indirection operator (*) in front of the 
name of the variable that you want to pass by pointer in the function declaration: 
 
void GetData(int *x, int *y) // use indirection to de-reference inside function 
 
Note that the call to the function requires the address-of operator in front of the corresponding 
variable is the function call. 
 
void GetData(&a, &b); 
 
Rules for Return Values 
Use of return values is perhaps a bit more complicated than pass by pointer. Specifically, there 
are three things you need to do whenever returning a value from a function. These are: 
 
1. Declare the function with the appropriate return type. 
That is, if the function returns a value then you must indicate what its data type is. For example: 
 
int Sum(int x, int y) 
 
float Average(int x, int y) 
 
Here the Sum() function returns an integer and the Average() function returns a float. 
 
2. Return a value of the appropriate type in the function. 
This means you need a return statement somewhere inside that function which has the 
values/variable/expression that you want to return inside brackets. For example: 
 
return(sum/2.0); 
 
This is the return statement from the Average() function and notice that the data type of the 
value returned is a float. Note usually the return statement is the last statement in the function. 
 
3. “Catch” the return value back in the calling function. 
For the Sum() function we can pass the return value straight to a call to printf(): 
 
printf(“The sum is: %d\n”, Sum(a, b)); 
 
However, more commonly the return value is assigned to a variable for future use: 
 
sum = Sum(x, y);

The way pass by value works is the reason we cannot use that mechanism for passing multiple 
pieces of data back from a function. However, there is another technique called pass by pointer 
which allows us to do this. 
 
Pass by pointer involves creating a pointer (of the same data type) to a variable which passes the 
address of the original variable into a function via the pointer parameter. 
 
This means if we change the value of something passed by pointer, the changes made inside the 
function are reflected (i.e. also occur) in the original variable that exists outside the function. 
 
C gives the programmer a lot of control over data passed by pointer, which makes it a very 
powerful feature. However, it also means it is possible to make mistakes and you need to be very 
careful. 
 
REMEMBER: if you use pass by value, a copy of the variables value (that exists outside the 
function) is passed into the function. It is possible to modify this value inside the function, 
however, any changes made to the parameter variable inside the function are not reflected in 
the original variable (that exists outside the function). If you use pass by pointer, the address of 
the variable (that exists outside the function) is passed into the function. Modifying this 
parameter variable is also possible inside the function; any changes made to the parameter 
variable are most definitely reflected in the original variable (that exists outside the function). 
This is a handy way to return more than one value from a function; it is not possible to do this 
with pass by value parameter passing.

Pass by Value 
So far, the parameter passing we’ve done has been one-way. The data has been passed into the 
function, but no data is passed out (except for the function’s return value which is not 
considered parameter passing). 
 
This type of parameter passing is known as pass by value. 
 
Pass by value works by making a copy of the variable being passed. 
 
For example the following line from the main() function involves passing two variables into 
Sum(): 
 
Sum(a, b); 
 
The function definition for the Sum() function looks like: 
 
int Sum(int x, int y) 
 
And we know that the parameter variables x and y will contain the values that were in a and b 
respectively. 
 
This is pass by value because x and y are the copies of a and b. They might have the same 
values, but they are different variables and are stored in separate memory locations. 
 
This means that if you make a change to the value of x inside Sum() it will NOT affect the value of 
a inside main(). 
 
So, the change is being made to the copy, not the original and once the function call ends the 
parameter variables disappear and the changes are lost.

Command Line Arguments 
In addition to reading in data while the program is running, programs can also be pre -supplied 
with data via the command line. Command line arguments are supplied to a program when that 
program is invoked on the command line. 
 
This means the program can immediately begin doing its job without waiting for input from the 
user; this can be a very flexible way of interacting with a program. 
 
Command line arguments are implemented in C using two different parts:  
1. argc: is the argument counter (how many command line arguments were supplied)  
2. argv: is an array of strings, each element of which is a separate command line argument  
 
Both argc and argv are parameters to the main function (see example below). As such they 
allow a program to access the command line arguments so that the program can utilize any such 
input. 
 
A simple example of how to use command line arguments in C is giv en below: 
 
#include <stdio.h> 
 
int main(int argc, char *argv[]) 
{ 
int i; 
 
if(argc == 1) 
{ 
printf("No command line arguments given!\n"); 
} 
else 
{ 
printf("Arguments are:\n\n"); 
 
for(i=0;i < argc;i++) 
printf("%d. %s\n", i, argv[i]); 
} 
 
return(0); 
} 
 
Examples: 
 
cmdline 
No command line arguments given! 
 
cmdline abc 
Arguments are: 
0. cmdline.exe 
1. abc 
 
cmdline abc def ghi 
Arguments are: 
0. cmdline.exe 
1. abc 
2. def 
3. ghi 
 
Note: in the example output above, the full path to the program has been omitted which will 
appear if you run it under Windows.

Passing Strings as Parameters 
We can see from the previous sections on strcpy(), strcat(), and strcmp(), that these library functions 
take strings as their arguments. This means that their formal function declarations must have 
formal parameter lists that include string parameters. So, how can we declare our own functions 
to take strings as parameters? 
 
As a string in C is a just character array, the best way to pass a string into a function is as a character 
pointer. It is also a very good idea to pass the length of the string into the functio n as well, so that 
you only process the correct amount of data stored in the array. 
 
Example program that reads a string from keyboard and swaps any occurrence of ‘a’ with ‘z’ : 
 
#include <stdio.h> 
#include <string.h> 
 
const int SIZE = 128; 
 
int readString(char *str) 
{ 
int len; 
 
printf("Enter a sentence:\n"); 
if (fgets(str, SIZE, stdin) != NULL) 
{ 
str[strlen(str) - 1] = '\0'; 
} 
len = strlen(str); 
 
return(len); 
}

Strings 
You may have noticed that so far in this unit we have carefully avoided one form of data that is 
actually very common. 
 
While computer programs deal easily with numbers, because a lot of the data they process 
relates to humans, they also deal with our languages. Strings are the data type for doing this. 
 
A string is simply a sequence or “string” of characters of a given length. These characters can be 
alphabetical, numbers (as characters – not numerical), and other symbols. 
 
Specifically, a string can be zero or more characters in length. Although very common, different 
programming languages handle strings quite differently. 
 
In C, strings are usually implemented (more or less) simply as an array of  characters (data type 
char). 
 
Storing Strings in C 
Since a string in C is stored as a character array, the length of this array can be defined statically. 
 
This means that you need to decide the maximum number of characters the string will store 
when writing the program. However, the actual string you store does not need to use up all 
available array elements. 
 
So, the length of the string may be less than the actual size of the array it is stored in. 
 
Because of this, it is necessary to mark where the actual end of the string is within the array. C 
does this using a special character called a null terminator. 
 
The null character is essentially binary 0, has no printable representation, but can be 
represented using the symbol '\0'. This is similar to the newline character '\n'. 
 
C library functions that process strings therefore look for the null character and use this to 
determine where the end of the string is. We therefore say that strings in C are null terminated. 
 
There are a couple of additional issues in relation to null terminated strings that need to be 
noted here. 
 
Firstly, while functions processing strings can look for the null character to mark the end of the 
string, unless explicitly given this information, these functions do not know the size of the array 
in which the string is stored. 
• They will often keep processing until they find the null character. 
• It is therefore important to make sure that you stay within the bounds of the 
array. 
 
Secondly, when creating a char array in order to store a string, you need to allow for storing the 
null character. For example, the following declaration can only store a name that is a maximum 
of 9 characters long (NOT 10) because one character needs to be left to store the null  character. 
 
const int SIZE = 10; 
char name[SIZE] = {'\0'}; // null terminator stored at index 9

Passing Arrays as Parameters 
As an array is a complex data type, we cannot pass it into a function (via the formal parameter 
list) using the Pass By Value (PBV) mechanism. Thus, an array is always automatically passed 
into a function using the Pass By Pointer (PBP) mechanism. This mechanism passes the address 
of the array (actually the address of the ﬁrst element of the array) into the function. 
 
The reason for automatically passing by pointer is because passing the address of an array 
allows for multiple data (stored in an array) to be passed into a function as one variable (the 
array name). This then provides the means to access all elements of the array inside the 
function, by using its index numbers. 
 
Thus, passing by pointer avoids the need to duplicate often huge amounts of data as would 
happen if using passing by value to pass multiple data into a function via individual variables. 
 
You can pass arrays into functions (via the formal parameter list) using either the square bracket 
notation or the indirection operator (both use the PBP mechanism). In the call to such a function, 
the array name can be used without the address-of operator in the argument list. 
 
const int SIZE = 10;

int sumArray(float *farr, int size) { … }

// Call to function sumArray (assumes array is populated) 
float total = sumArray(floatArray, SIZE); // no & operator 
 
return 0; 
} 
 
Because array passing is by pointer, when you pass an array into a function any values you put 
into the array remain there when the function returns. Also, any values you modify in the array

Bounds Checking 
Creating an array gives you an area of memory to store as many pieces of data as you specify.  
Some languages keep track to make sure you stay within this area — this is called bounds 
checking. However, C does not do this. 
 
This can lead to some subtle errors where your program attempts to access array elements that 
don't exist. These errors will likely lead to your program behaving strangely, or very probably 
crashing. This most commonly happens when you forget that array numbering begins at 0. 
Examples for an array declared as: 
 
const int SIZE = 20; 
int nums[SIZE]; 
 
nums[20] = 5;    // index numbers are from 0 - 19 
 
OR: 
 
for(i = 0;i <= SIZE;i++) // index numbers are from 0 - 19 
{ 
printf(“Enter element %d: ", i); 
scanf(“%d%*c”, &nums[i]); 
} 
 
Both of these errors occur because the program attempts to access an element one passed the 
end of the array. 
 
The graphic below emphasises how a computer has memory that is free for you to us e, and 
memory that is restricted for other programs, operating system, etc.  
 
When you create a variable in your program, you are asking the computer to give you a certain 
amount of memory to store data in it. After data is stored in memory, you are not allowed to 
access any other memory beyond that memory, because you could be tampering with other 
restricted data. This is particularly true with arrays since you can access the array elements using 
the index, which cannot be larger than the size of the array.

Introducing Arrays 
Data types are critically important in that they determine the properties of the variables and 
values being processed. All of the data types we have dealt with so far have been built into the 
language. For example, char, int, and float are all part of C. 
 
These built-in types are called simple or primitive data types. In general, these cover the majority 
of different types of data that we need to store and process in our programs.  However, 
sometimes they are not enough; there are situation that require the ability to create our own 
data types. These new types are no longer built-in, they are called complex data types. 
 
Complex data types are created by joining together the simple data types that are built into the 
language. In this case, complex means made up of multiple parts, rather than difficult. In this 
topic we will be start looking at one of the major complex data types: the array. In a later Module 
we will learn another way to define our own data type (using C structs), and in future units you 
will learn still other mechanisms for creating your own data types. 
 
What are Arrays? 
So far, we have made programs that contain a small number of variables. However, if we have 
lots of data to store, this soon becomes impractical. 
 
An array is a complex data type that groups together a number of different values  of the same 
data type into a single structure. 
 
The advantage of using an array is it becomes easy to create algorithms to solve problems that 
otherwise would be very difficult and time-consuming to deal with. 
 
An array is defined as a contiguous block of memory holding n items, called elements, of a given 
data type. Each of these elements is assigned a number to identify its position within the array. 
This number is called an index. In C arrays, the index values start at 0. 
 
For example, the following is a pictorial representation of an array storing 10 integers: 
 
Note: the index numbers are sequential (starting at 0), but the values stored at each index can 
be any integer; there need not be any order to them. 
 
An array is given a name, and this can be used just like any other variable name, except it must 
be used with its index number in order to access its elements (more soon). 
 
Array Advantages 
So instead of having lots of separate variables, each storing an individual value, we can have a 
single variable storing many pieces of data. There are a number of advantages to this.

2.2 Algorithm
Think of modular design as a way to make the design process more manageable, by
decomposing a large problem into smaller problems and solving the smaller problems
one at a time. It involves identifying themajor steps that will completely and correctly
solve a given problem, and specifying them in the High Level Algorithm (HLA)1. In the
following example, input into a program is required so amodule (getInput) is speciﬁed
in the HLA; two other modules are also speciﬁed.
High Level Algorithm (HLA)
// call module to receive user input
getInput()
// call module to process the user input
processInput()
// call module to present output
display()
This means that these modules will be ‘called’ to perform their designated tasks.
There is no need at this point to deﬁne the particular instructions that each module will
perform; you only need to know that the modules do certain tasks, and that they are
‘called’ when those tasks are required to be performed. The detailed deﬁnition of each
module’sinstructionscanbecompletedatalatertime(whenrequired). TheHLAmustbe
complete, and should consist of predominantly module calls, with each module speciﬁed
to perform a speciﬁc task; all modules together must provide the complete solution (but
remember, the low level details are not deﬁned at this stage). Control structures such as
loops and ‘if’ statements are permitted in the HLA (if appropriate), but the HLA should
be kept mainly to module calls unless control structures are necessary.
Once the HLA is complete, the detailed instructions for each of the modules speciﬁed
in the HLA must be deﬁned in the Low Level Algorithm (LLA). That is, each module def-
inition in the LLA should precisely express the detailed processing involved in performing
its designated task.
When developing your LLA, deﬁne the instructions for one module until it is complete
and correct. Then deﬁne the next module until it is complete and correct, and so on until
all modules are deﬁned. Module deﬁnitions may involve sequential instructions and/or
control structures or even a call to another module.

2 Modular Design
2.1 Principles of Modular Design
When designing a solution using modular design, carefully consider the four principles:
1. Abstraction: means hiding details that aren’t necessary to know, while understand-
ing in general terms what needs to be done. In terms of modular design, a High
Level Algorithm (HLA) allows us to do this. We can specify what major tasks are
required to completely solve a problem, but leave the detailed instructions until
each speciﬁc module/function is deﬁned. And even when developing the speciﬁc
instructions for one module/function, the details for another module/function do
not need to be deﬁned until working on that module/function.
2. Code Re-use: If a module/function already exists to solve a particular problem or
perform a particular task, why develop and write your own version of that mod-
ule/function? This is particularly true if the existing module/function has been ex-
tensively debugged and tested. C libraries contain such existing modules/functions.
You have been using some (eg: printf() and scanf()). In terms of modular design,
you should look for situations where a task can be deﬁned in one module/function
and then re-used, if that task is required again somewhere else in your design.
3. High Cohesion: refers to the cohesiveness of your design. Any particular mod-
ule/function should be responsible for performing one task, and one task only. A
module/function that does a number of tasks (which aren’t all related to the same
task) is not cohesive and becomes too general, less usable, and harder to maintain.
The combination of module/function calls in your overall design, is responsible for
the cooperative functionality of all modules/functions, which must be combined in
such a way as to perform the complete functionality of your solution.

Generating the Numbers 
The rand() function generates a number (semi-)randomly chosen between the ranges of 0 and 
the built-in constant RAND_MAX. 
The following program demonstrates this behaviour: 
 
#include <stdio.h> 
#include <time.h> 
#include <stdlib.h> 
 
int main() 
{ 
int r; 
 
srand(time(0)); 
r = rand(); 
 
printf("Random number chosen between zero and %d: ", RAND_MAX); 
printf("%d\n", r); 
 
return(0); 
} 
 
However, most of the time it is more useful for the programmer to specify the maximum 
random number to be obtained. 
 
Limiting the Range 
The modulus operator can be used to limit the maximum value obtained when generating a 
random number. 
• Modulus gives the remainder left over after division. 
• The remainder can be between 0 (if it goes in evenly) or, at most, one less than that 
number. 
 
Therefore, calculating the modulus of a random number by the largest possible value you would 
like to get back (max) will give you a number in the range: 0–(max-1). 
• Adding one to this result will give you a number between 1 and max. 
 
The following program illustrates this technique using a maximum value provided by the user: 
 
#include <stdio.h> 
#include <time.h> 
#include <stdlib.h> 
 
int main() 
{ 
int max, r; 
 
printf("Enter largest possible number: "); 
scanf("%d%*c", &max); 
 
srand(time(0)); 
r = (rand() % max) + 1; 
 
printf("Random number chosen between 1 and %d: %d\n", max, r); 
 
return(0);

The C Approach 
The way C approaches this is to use two separate files when writing a library. One file contains 
the (private) implementation and this gets compiled in a special way and becomes the library 
object code that is linked together with other program code. 
 
The other file is the interface, and this is called a header file and ends with the extension .h 
 
These are the files that are referenced in the #include directives at the start of most C programs. 
 
#include <stdio.h> 
 
The header file itself doesn’t contain any real code (implementation), just a set of definitions 
stating what functions are part of library, what parameters they take and values they return. 
 
It also usually contains extensive comments describing what each function does and what its 
parameters are. This effectively defines the interface for those library functions. 
 
Although we won’t be creating any libraries in this unit, we will be using modules from such 
libraries so these concepts are important to understand. 
 
Using Built-In Functions 
Random Number Generation 
The generation of random numbers is a relatively common task in many programs. However, 
most modules for this purpose actually generate pseudorandom numbers. These numbers are 
not truly random and are potentially predictable, so should not be used in situations where 
unpredictability is important. 
 
Computers can't simply pick random numbers -- they can only “calculate” them. However, this 
means that if an algorithm for calculating random numbers is performed, the result will be the 
same every time. 
 
To avoid this, the random number generator must be “seeded”, basically by giving the computer 
a random value on which to base the sequence it calculates. 
  
In C, the random number generator can be seeded using the srand() function. However, this 
function must be given a value as the seed. The easiest way to do this is to use the time, which is 
constantly changing and therefore will normally give different results each time: 
 
srand(time(0)); 
 
Note that seeding should only be done once at the start if generating a sequence of numbers in 
a program. Otherwise the same number will be generated.

Advanced Modular Programming Concepts 
Public Interface and Private Implementation 
Public Interface 
The modules that you write may become part of a library and be used by lots of different 
programmers in solving lots of different problems. As a result, the details of exactly how they are 
designed matter a lot. 
 
One important aspect of this is the first principle of modular programming that we looked at: 
abstraction. 
 
That is, just because a programmer is going to use a module that someone else has written, it 
doesn’t mean they necessarily need to know exactly how that module works. Instead they just 
need to know what it does and how to use it. 
 
This is known as the interface of the module. In fact, libraries are often described as application 
programming interfaces or APIs. Because the interface to a module needs to be known in order to 
use it, this part of the module is described as being public. 
 
Private Implementation 
So long as you know a module’s interface, you don’t need to know the details of exactly how it 
works (abstraction). 
 
So, it is often said that the details of how a module works are private. This implies that they are 
virtually hidden from the programmer who is using the module and to some extent this is true. 
 
The code that actually makes up the module is often described as its implementation. 
 
The implementation of a module is often kept separate from its interface in order to enforce the 
separation between public and private. This helps to enforce the principle of abstraction. 
 
As long as the implementation is consistent with the described interface then this does not 
matter.

Code Re-use and Libraries 
Code re-use was mentioned as an important principle applying to modular programming. This 
means that when you write a module, ideally its interface should be general enough that it can 
be re-used. 
 
High cohesion, low coupling and abstraction help to facilitate this. However, while the module 
should solve the problem in a general way to facilitate re-use, it should also maintain high 
cohesion. 
 
However, module re-use is not just limited to within the same program. Often modules are 
general enough and solve a common enough problem that it is likely they could be used again in 
other programs. 
 
This leads to the collection of modules together into groups called libraries which can then be 
used in many programs. This saves the programmer re-writing code that they or someone else 
has already written (and which has most likely been thoroughly tested). 
 
Most programming languages come with extensive libraries for various purposes such as: 
• Character and “string” manipulation. 
• Input/output: both to files and interactively (i.e. keyboard and screen). 
• Graphics manipulation and user interfaces. 
 
For example, printf() and scanf() are part of the Standard C Library. 
 
 
Summary of Modular Programming Concepts 
Here are the modular programming concepts we have covered so far: 
• We can use top-down design where we start with a high-level (general) view of the 
solution and fill in the gaps, step by step. 
• This process of starting with a general solution and refining each step to work out the 
details makes it much easier to develop correct solutions to complex problems. 
• This approach relates closely to modular programming where we split up the code into 
separate parts, each of which solves a specific part of the problem. 
• These modules (called functions in C) should all adhere to the principles of abstraction, 
code re-use, high cohesion and low coupling. 
• Functions can either produce a result (returning a piece of data of a given type) or just 
perform an operation (return void). 
• We can pass data into a function and also pass data back out as well (in addition to 
returning a single value). 
• If the data only needs to go into the module then it is passed by value and this involves 
making a copy of the data: any changes are made to the copy but not the original. 
• Alternatively, data can be passed by pointer where the address to the original data is 
passed into the module: changes are therefore made to the original and so these persist 
after the function has returned.

Or alternatively, it could be used in an expression as if it were a variable or a constant. For 
example, the last two lines of the Average() function are as follows: 
 
sum = Sum(x, y); 
return(sum/2.0); 
 
But these could just as easily have been condensed into a single expression on one line: 
 
return(Sum(x, y)/2.0);

returns. Always remember to do something with this value, otherwise it is simply thrown away 
and your program continues executing. 
 
This is a very common error by beginner programmers! 
 
So, when should you pick return values over pass by pointer? 
 
If a function produces more than one value to be returned, then pass by pointer must be used. 
But this doesn't mean it has to be used for all of these values and it may be appropriate to also 
use a return value for one of them. Similarly, just because just one piece of data comes out of 
the function does not mean that you have to use return values (though it is generally best 
practice). 
 
If a function produces a single value, then it may be appropriate to use return values instead of 
pass by pointer. This is particularly so when this value is the result of some calculation. This is 
probably the only clear rule that can be applied. 
 
Summary of Passing/Returning Values Into/Out of Functions 
There are ways to pass/return values into/out of a function: 
1. Out of only, using the function return value. This mechanism is convenient, and ideal for 
situations where the function needs to return one value only. 
2. Into only, using pass by value parameter variables. This mechanism allows values to be 
passed into a function via parameter variables, for use within the function. Modification 
of any parameter variable is allowed, but any changes to a parameter variable is not 
reflected in the original variable (outside the function). 
3. Into and Out of, using pass by pointer parameter variables. This mechanism also allows 
values to be passed into a function via parameter variables, for use within the function. 
Modification of any parameter variable is allowed, and any changes to a parameter 
variable is reflected in the original variable (outside the function). This then provides the 
means of returning multiple pieces of data from a function.

Pass by Pointer vs Return Values 
Pass by pointer and return values do similar things: 
• Both allow you to get data out of a function, which would otherwise be impossible due to 
the limitations on variable scope. 
• However, these are two completely different techniques. 
• Even though the result is the same, the way they work is quite different. 
• Also remember that a function can only return a single value (by its return statement) 
whereas as many parameters as you like can be passed by pointer. 
 
There are also significant differences in how you use each of these features. 
 
Rules for Pass by Pointer 
With pass by pointer it is very easy. You simply put an indirection operator (*) in front of the 
name of the variable that you want to pass by pointer in the function declaration: 
 
void GetData(int *x, int *y) // use indirection to de-reference inside function 
 
Note that the call to the function requires the address-of operator in front of the corresponding 
variable is the function call. 
 
void GetData(&a, &b); 
 
Rules for Return Values 
Use of return values is perhaps a bit more complicated than pass by pointer. Specifically, there 
are three things you need to do whenever returning a value from a function. These are: 
 
1. Declare the function with the appropriate return type. 
That is, if the function returns a value then you must indicate what its data type is. For example: 
 
int Sum(int x, int y) 
 
float Average(int x, int y) 
 
Here the Sum() function returns an integer and the Average() function returns a float. 
 
2. Return a value of the appropriate type in the function. 
This means you need a return statement somewhere inside that function which has the 
values/variable/expression that you want to return inside brackets. For example: 
 
return(sum/2.0); 
 
This is the return statement from the Average() function and notice that the data type of the 
value returned is a float. Note usually the return statement is the last statement in the function. 
 
3. “Catch” the return value back in the calling function. 
For the Sum() function we can pass the return value straight to a call to printf(): 
 
printf(“The sum is: %d\n”, Sum(a, b)); 
 
However, more commonly the return value is assigned to a variable for future use: 
 
sum = Sum(x, y);

Pass by Pointer 
The way pass by value works is the reason we cannot use that mechanism for passing multiple 
pieces of data back from a function. However, there is another technique called pass by pointer 
which allows us to do this. 
 
Pass by pointer involves creating a pointer (of the same data type) to a variable which passes the 
address of the original variable into a function via the pointer parameter. 
 
This means if we change the value of something passed by pointer, the changes made inside the 
function are reflected (i.e. also occur) in the original variable that exists outside the function. 
 
C gives the programmer a lot of control over data passed by pointer, which makes it a very 
powerful feature. However, it also means it is possible to make mistakes and you need to be very 
careful. 
 
REMEMBER: if you use pass by value, a copy of the variables value (that exists outside the 
function) is passed into the function. It is possible to modify this value inside the function, 
however, any changes made to the parameter variable inside the function are not reflected in 
the original variable (that exists outside the function). If you use pass by pointer, the address of 
the variable (that exists outside the function) is passed into the function. Modifying this 
parameter variable is also possible inside the function; any changes made to the parameter 
variable are most definitely reflected in the original variable (that exists outside the function). 
This is a handy way to return more than one value from a function; it is not possible to do this 
with pass by value parameter passing.

and the address-of operators. 
 
When defining a function that passes the parameters by pointer, the indirection operator must be 
used in the formal parameter list:

Pass by Value 
So far, the parameter passing we’ve done has been one-way. The data has been passed into the 
function, but no data is passed out (except for the function’s return value which is not 
considered parameter passing). 
 
This type of parameter passing is known as pass by value. 
 
Pass by value works by making a copy of the variable being passed. 
 
For example the following line from the main() function involves passing two variables into 
Sum(): 
 
Sum(a, b); 
 
The function definition for the Sum() function looks like: 
 
int Sum(int x, int y) 
 
And we know that the parameter variables x and y will contain the values that were in a and b 
respectively. 
 
This is pass by value because x and y are the copies of a and b. They might have the same 
values, but they are different variables and are stored in separate memory locations. 
 
This means that if you make a change to the value of x inside Sum() it will NOT affect the value of 
a inside main(). 
 
So, the change is being made to the copy, not the original and once the function call ends the 
parameter variables disappear and the changes are lost.

we will introduce the basic concept of pointers. Note we will primarily discuss pointers in relation 
to parameter passing. 
 
Recall from Module 2 lecture notes (the section on variables), we said the variables are a way to 
store and manipulate data; this allows a program to access/modify the data during execution of 
the program. We said that a variable is declared of a certain data type, according to what type of 
data will be stored it. 
 
The figure on page 6 of Module 2 lecture notes demonstrates a “theoretical” memory 
configuration. It shows 3 variables linked to memory cells that store data of the correct type for 
those variables. A program can access the memory locations via the variable names. 
 
A computer program must know the address of memory locations in order to be able to access 
these locations via variable names or other means. The name given to a program’s memory area 
(in relation to temporary storage during program execution) is RAM (random access memory). It 
must have a starting and finishing address and be allocated in logical physical sized blocks.

Variable Scope 
Remember the principles of abstraction and low coupling to which modules should adhere? This 
also applies to data (i.e. the variables being used), but in a special way. Variables cannot be 
“seen” (i.e. their values accessed and changed) everywhere throughout the program. In fact, 
normally variables are only visible within the function in which they are declared. 
 
This is called the variable’s scope. 
 
So the variables a, b and total from the main() function in the previous program are not visible 
inside the Sum() function. 
• This is why we can’t just write return(a+b) inside the Sum() function (it would also 
demonstrate poor abstraction and high coupling). 
• Only the formal parameters (x and y) are visible within the Sum() function (passed in via 
the parameter list). 
 
By limiting the scope of variables to the function in which they are declared we can enforce 
abstraction (other functions don’t know what data is inside one another and they don’t need to 
know) and low coupling (other functions cannot accidentally change the data within a function.) 
 
Also because of variable scope, the way parameters are passed into and out of a function (plus 
any return values) becomes very important. These define the interface to that function.

Modularisation in C 
Function Syntax 
In virtually all programming languages there is some way of splitting up the body of code that 
makes up the program into separate modules. However, there are different philosophies and 
paradigms for doing this. 
 
In C, these modules are called functions. 
 
Each of these functions performs a specific task related to solving the overall problem. This 
directly relates to the notion of top-down design. The design of the functions that make up a C 
program should adhere to the principles of abstraction, code re-use, high cohesion and low 
coupling. 
 
We have actually already been using functions in our C programs! 
 
All programs must have a function called the main function and this is where execution begins. It 
is called by the operating system, which is why it is always the first code executed. Our programs 
so far have only had a single function, namely the main() function. 
 
Programs that are made up of other functions still always have a main() function. However, the 
job of the main() function is no longer to contain all the code to solve the entire problem. 
Instead it now calls the other functions which perform the individual sub-steps that make up the 
algorithmic solution. This means the main() function effectively becomes the 
implementation of our high-level algorithm with the other functions representing the 
refined steps.

Code Re-Use 
The concept of code re-use is summed up in a common programmer’s catch-cry: 
 
Don’t re-invent the wheel! 
 
If code has already been written to solve a particular problem, why develop and write your own 
version of that code? This is particularly true if the existing code has been extensively debugged 
and tested. 
 
Code re-use also applies to the concept of modularity. A single program will often use one of its 
modules a number of times for slightly different purposes. For this reason, it is best to make 
modules as general as possible in their behaviour providing it still solves the original problem 
and does not become much more complex. 
 
For example, if you need a module to calculate the average test marks for a class of 10 students, 
it would be better to write the module so it will work for any number of students. That way, 
if/when the class size changes, you can re-use the module. 
 
Re-use also applies to so-called “code libraries” which are collections of modules made available 
to all programmers working with a particular language/platform. For example, the printf() and 
scanf() functions we have been using are modules provided as part of the C library. 
 
High Cohesion 
The code which makes up a module should exhibit high cohesion. What this means is that all of 
the code in that module should be concerned with the same task (and only that task). 
 
A module that does a number of tasks (which aren’t all related to the same sub-step) is not 
cohesive and becomes less general and harder to maintain. 
 
This is because the code associated with a specific task will no longer be isolated to a single 
module but possibly spread over many separate modules. 
 
So all the code in a module should be associated with the same logical task. 
 
Low Coupling 
Low coupling between modules means that each module should have limited ability to 
interact with data belonging to other modules. 
 
If a module can alter the data belonging to another module then this could cause that module to 
fail and the bug would be very hard to find. 
 
Low coupling is closely related to abstraction since hiding the data associated with a module 
helps to abstract the behaviour of that module.

Abstraction 
Abstraction is the process of hiding details that are not necessary for us to know while 
understanding in general terms what is going on. 
 
For example, most people drive a car, but few are mechanics. 
 
We know that our cars have engines which consume fuel when we drive; and we need to re-fuel 
regularly otherwise our cars will stop working. But we don’t need to understand the details of 
what is actually going on inside our car’s engine when it burns the fuel. Instead we just focus on 
the general things that we need to know to do what we need to do. 
 
Therefore, we abstract the details of how our cars work. 
 
The same applies when we split up a problem into steps/modules and then refine the details of 
exactly how each of those modules performs its task. Although all the modules must work 
together to solve the problem, no individual module should need to know exactly how another 
module works. 
 
Instead a module just needs to know in general terms what the other module does (primarily 
what its outcomes are), plus any data that it requires to perform its task. 
 
In general, as little information as possible about how a module works (as opposed to what it 
does or its outcomes) should be available outside of that module. (Of course, the developer of 
the module must understand the low-level details of what a module does)

Top Down Design 
Most programmers and computer scientists would probably agree that programming is hard. 
This is not so much that it is hard to learn (although you might think it is right now!), rather that 
writing a program which solves any non-trivial problem and works reliably (without bugs) 
is very difficult. 
 
The main reason for this is that, for all but the simplest programs, a lot of complexity is required 
to provide a comprehensive piece of software. In fact, computer programs are among the most 
complex and sophisticated things that humankind has ever created. 
 
Computer scientists want to make this process easier and so a lot of work has gone into 
reducing the complexity. If computer code can be made more manageable then it will be 
easier to write programs and will have fewer bugs. Just as importantly, it will also be 
easier to fix any bugs that are found. 
 
Taking it from the Top 
Top down design (TDD) is one way of trying to make the complexity more manageable. 
 
TDD means that we approach the task of solving the problem (and writing the code) starting 
from the top (the high-level details) and working our way down to the bottom (the low-level 
details). 
 
In Module 1 we developed a high-level algorithm and gradually refined it to the individual steps 
needed to solve the problem. This is exactly how TDD works. 
 
So TDD involves:  
1. Developing a high-level algorithm containing a number of steps which will solve the 
problem. These should be general steps and will not usually involve the specific 
calculations needed. 
2. Taking each of these steps and decomposing them individually. In other words, working 
out what more detailed, lower-level steps need to be performed to complete the higher-
level step. This is often called refining your high-level solution. 
 
Sometimes you will have to have several layers of decomposition/refinement, depending on the 
complexity of the problem. 
 
Modularity and Top-Down Design 
So how does this process of breaking down the solution into more detailed, lower-level steps 
actually work? In practice this is closely linked with the notion of modularity. 
 
Modularity means that each of the refined steps is separated out into a distinct module of code. 
 
Each of these modules performs each of the refined sub-steps from the high-level algorithm. 
However, for TDD to be successful in terms of providing a useful methodology for dealing with 
complexity, these modules must be constructed in a certain way. 
 
Specifically, they should have a number of important properties: 
• Abstraction between modules.
