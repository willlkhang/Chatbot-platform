Q:
For my date and time classes I have kept the input and output operators in date.h and time.h. You had told me to remove them as putting the operators in the .h files violated MVC. I think I understood what you told me, but now I think I am not so clear as I didn't write it down.

I/O is tied to formatting. That is what the input data or output data looks like. It is part of the "View". The model is the actual data class, Date and Time for this work. The assignment in this unit is tiny compared to code bases out there, but the principles still apply and you must always follow the principle where you minimise coupling. All this was discussed in the workshops. 

If you keep the View with the model, you will get no marks. Not getting marks is really a small price to pay compared to what would happen with industry code where a small change in the View in a poorly designed code base results in changes proliferating throughout the entire code base (Jelly effect referred to in the PowerPoint slide and workshops), leading to bugs, cost overruns, delay in delivery.

Lab 4 was about removing the View, including the formatting concerns from the Model, and the lessons must be utilised and built upon moving forwards.

Minimising coupling and increasing cohesion rules must never be violated. You would know this from ict159 last year where a mark of 0 would be given for the entire assignment if modularity rules were not followed. This is coming from industry as we become more industry focussed. 

So do not violate SOLID. The cost is heavy and it does not matter if the code works now. The problem will occur later.

Q
For this particular solution I no longer use variable length arrays, and just expose the internal array for my vector class via a public function (returning a T const *).

A:
No it is worse than that. If later on, you change the implementation to some other data structure which does not use contiguous storage in memory, every client program that uses your data structure will break.

The designer of a data structure should never allow a coupling between a client program and the internals of a data structure. All the client program knows is the interface for the data structure. See the PIMPL idiom in the coding standards book that was pointed out in one of the workshops. Although the advise is to use this idiom judiciously, the reasons that argue its usage should be taken to heart even if the PIMPL idiom is not used.

You should be careful that a getter should not allow modification of the data. A getter is meant to just return data, not modify it. See rule 15 Use const proactively in
"C++ coding standards: 101 rules, guidelines, and best practices". The getter should also prevent itself from modifying class data by having const on the right hand side.
 int GetCredits() const;
It is the setters job to modify data, not the getters, so don't mix behaviour as you can loose all the code marks for providing unsafe code. This idea applies to any code you write not just C++ code.

Q:

Q:
I have decided to use setAt and getAt for the Vector instead of []. Would this be ok?

Are [] more powerful than normal methods like getAt or setAt?

A:
Examine the specifications of Vector carefully. It is just an encapsulated raw array and thus Vector should behave like a normal array but with certain protections. To behave like a normal array, the interface must be that of an array. If you change the interface then you have changed how the user interacts with it.

So think very carefully, and provide your reasons in the data dictionary table (Lec 11. ppt of Topic 4).

I am not sure of the context of "powerful" that you are referring to.
op[] has the intuitive usage in the array context. It takes an index
and returns an element. A bit more on this in the lab5 QandA. It is
no different to a subroutine call. 

As an overloaded operator, you shouldn't change the semantics (meaning) of the operator. What does rule 26 say about operators? See the unit reference book C++ Coding Standards ebook from the topic readings.

A normal routine can have a number of parameters and can return data back like functions. So [] can't do everything compared to a normal routine like setAt or getAt, but [] is more intuitive usage for arrays or data stored in consecutive memory locations and indexing makes sense.

The Vector you have to use is an encapsulated array, so you will need to justify why your removed conventional array behaviour and violated Vector requirements.

So the choice is really up to you and how you justify your choice. Your justification may not be accepted if you not put careful thought into the decision.

Q:
For Assignment 1 should we use names similar to the standard vector 
class for functions  so that their purpose is clear?

A:
The std vector is from STL. It is bloated where functionality is 
duplicated in some of the methods. Naming conventions are used that 
are borrowed from other data structures like the stack (push, pop).
This is totally not helpful from the point of view of abstraction.
A Vector is not a stack, so must *not* pretend to be one. A Vector 
is a more generic linear data structure.

Your Vector needs to take a minimalist approach. Avoid duplication of 
functionality and don't mix names with other more specialised data 
structures like stacks and queues.

The Vector for the assignment is **not** the same as the STL vector.

The entire Specification and Implementation of the Vector for the assignment is about an A4 page of code laid out neatly without the doxygen comments. Also see Lab 5 QandA.
